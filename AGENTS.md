# AquaFlow Agent Notes

This file gives AI coding agents the project map, current architecture, and working rules for this repository.

> Read this file first on every task. It is meant to save you from re-exploring the codebase. If something here is wrong or outdated, fix it as part of your change.

## Project Layout

Everything lives under the `AquaFlow/` directory at the repo root:

- `AquaFlow/AquaFlow.sln` - solution file.
- `AquaFlow/AquaFlow.WebAPI` - ASP.NET Core Web API host, controllers, JWT auth wiring (`Program.cs`), `Filters/ExceptionFilter`, and `Services/AccessManager`.
- `AquaFlow/AquaFlow.Services` - service/business logic layer, validators, EF Core `DbContext`/entities/migrations.
- `AquaFlow/AquaFlow.Model` - shared request/response DTOs, search objects, `Access` DTOs, exceptions, and simple contracts.
- `AquaFlow/AquaFlow.Common.Services` - cross-cutting services; currently `CryptoService` (`ICryptoService`) for password hashing.
- `AquaFlow/AquaFlow.Services.Tests` - xUnit test project for the service/business logic layer (EF Core InMemory provider). Run with `dotnet test .\AquaFlow\AquaFlow.Services.Tests\AquaFlow.Services.Tests.csproj`.
- `AquaFlow/AquaFlow.WebAPI.Tests` - xUnit test project for controllers. No database or HTTP host: tests new up a controller directly with a hand-written fake of the relevant `IBaseCRUDService<...>` (see `UserNotifications/FakeUserNotificationCrudService.cs`) and a `ClaimsPrincipal` built with the claims under test, then call the action method and assert on the returned `ActionResult`. This only exercises code inside the action body - `[RequirePermission(...)]` runs as an MVC authorization filter, not a plain method call, so it isn't reachable this way; where a test needs to pin that a write action is still gated, it asserts the attribute (and its permission code) via reflection instead of invoking the pipeline. Run with `dotnet test .\AquaFlow\AquaFlow.WebAPI.Tests\AquaFlow.WebAPI.Tests.csproj`.
- `AquaFlow/UI` - Flutter client (`aquaflow_desktop`), a cross-platform (desktop/mobile/web) frontend for the Web API. See the "UI (Flutter client)" section below.

## Runtime

The Web API targets `.NET 9`. All persistence is SQL Server through EF Core. There is no in-memory data store anymore (`AquaFlow.Services/InMemory` has been removed).

`Program.cs` throws at startup if either of these is missing, so they are required to run the API:

- `ConnectionStrings:DefaultConnection`
- `JwtToken:Issuer`, `JwtToken:Audience`, `JwtToken:SecretKey` (optional `JwtToken:DurationInMinutes`, default 60)

For convenience in this test project, `appsettings.json` now ships **dev-only** defaults for both the connection string and the JWT settings, so the API runs out of the box without extra setup. These are throwaway local values (same SA/demo password as the Docker container); do not treat them as real secrets and always override them via environment variables or user secrets outside local dev. The environment variables below still take precedence over `appsettings.json`, so use them to run against a different DB or to supply a real secret:

```powershell
$env:ConnectionStrings__DefaultConnection='Server=localhost,1435;Database=AquaFlow;User Id=sa;Password=AquaFlow123!;TrustServerCertificate=True;Encrypt=False'
$env:JwtToken__Issuer='AquaFlow'
$env:JwtToken__Audience='AquaFlowClients'
$env:JwtToken__SecretKey='<any-local-dev-secret-at-least-32-chars>'   # do NOT commit a real secret
$env:JwtToken__DurationInMinutes='60'
$env:ASPNETCORE_ENVIRONMENT='Development'
```

Build the solution:

```powershell
dotnet build .\AquaFlow\AquaFlow.sln
```

Run the SQL Server container (maps host port `1435` to container `1433`; use `localhost,1435` from the host):

```powershell
cd .\AquaFlow
docker compose up -d
```

Run the Web API locally (set the env vars above first; the `http` profile alone does not set the connection string or JWT settings):

```powershell
dotnet run --project .\AquaFlow\AquaFlow.WebAPI\AquaFlow.WebAPI.csproj --launch-profile http
```

Apply EF Core migrations locally (needs at least `ConnectionStrings__DefaultConnection`):

```powershell
dotnet ef database update --project .\AquaFlow\AquaFlow.Services --startup-project .\AquaFlow\AquaFlow.WebAPI
```

Local URLs (http profile listens on `5161` on all interfaces for mobile LAN testing, https on `7286`):

- `http://localhost:5161` - redirects to the API reference in development.
- `http://<PC-LAN-IP>:5161` - same API from a physical phone/tablet on the same Wi-Fi.
- `http://localhost:5161/scalar/v1` - Scalar API reference UI.
- `http://localhost:5161/Access/login` - obtain a JWT (see Authentication).
- `http://localhost:5161/Users`, `/UserRoles`, `/Permissions`, `/UserRolePermissions`, `/WaterMeters`, etc. - resource endpoints (require a JWT).

When running with only `--urls http://...`, the HTTPS redirect middleware logs `Failed to determine the https port for redirect` and does not redirect, so plain HTTP works for testing.

## Authentication & Authorization

Authentication is implemented with JWT bearer tokens.

- `BaseReadController` is annotated `[Authorize]`, so every controller deriving from `BaseReadController`/`BaseCRUDController` requires a valid bearer token. Unauthenticated calls return `401`.
- Granular authorization is layered on top of `[Authorize]` with `[RequirePermission("Code", ...)]` (`AquaFlow.WebAPI/Filters/RequirePermissionAttribute.cs`). It is a `TypeFilterAttribute` whose nested `IAuthorizationFilter` returns `401` when unauthenticated and `403` (`ForbidResult`) when the caller holds none of the listed permission codes. Codes are matched case-insensitively against `Permission` claims. Apply it per-action (see `UsersController` overriding `Create/Update/Patch/Delete` with `Users.Manage`, `NotificationsController` overriding all six CRUD actions - including `GetAll`/`GetById` - with `Notifications.Manage`, making `/Notifications` an admin-only endpoint over the raw, unfiltered table; the self-service read path for an ordinary user is `GET /UserNotifications/mine` instead) or at the controller class level (see `PermissionsController`/`UserRolesController`/`UserRolePermissionsController` with `Roles.Manage`). The base CRUD write actions are `virtual` so derived controllers can override them just to attach the attribute; `BaseReadController.GetAll`/`GetById` are also `virtual` (return `ActionResult<...>`) so a resource can override reads too when self-service ownership filtering is needed. Other CRUD controllers still carry a `// TODO: add [RequirePermission(...)]` marker until their final codes are defined.
- `NotificationsController` never trusts a client-supplied `CreatedById` (`NotificationInsertRequest`/`UpdateRequest`/`PatchRequest` all carry it, since `AquaFlow.Services.Tests` constructs those requests directly against `NotificationService` and needs to set it): `Create` overwrites it with the caller's own id from the JWT `Id` claim before calling `base.Create` (same pattern as `AccountController.GetCurrentUserId()`), so a `Notifications.Manage` holder cannot author a notification under someone else's name. `Update`/`Patch` instead fetch the existing row via `Service.GetByIdAsync` and force the request's `CreatedById` back to the existing value, so authorship can never be reassigned by an edit either (mirrors the FE, which already round-trips the original `createdById` on edit in `admin_notifications_screen.dart`).
- `UserNotificationsController` (`/UserNotifications`) is a self-service inbox with an admin escape hatch, combining both patterns above: `Create/Update/Delete` are overridden with `[RequirePermission("Notifications.Manage")]` (admin/system-only; rows are normally created by `NotificationService.InsertAsync`). `GetAll`/`GetById` are overridden without the attribute: a caller holding `Notifications.Manage` passes through unmodified (admin listing), otherwise the caller's id from the JWT `Id` claim is force-applied (`GetAll` pins `search.UserId`; `GetById` 404s - not `Forbid`, to avoid confirming another user's row exists - when the fetched row's `UserId` doesn't match). `Patch` is overridden with no permission attribute at all (an owner must be able to mark their own `ReadAt`) but does the same ownership check plus a mass-assignment guard: if the caller lacks `Notifications.Manage` and the request sets `UserId` or `NotificationId`, it throws `ClientException` before touching the service, so a caller can never patch someone else's row or reassign their own row's ownership - only `ReadAt` is self-editable. `GetMine` (`GET /UserNotifications/mine`) predates this and uses the same JWT-id-pinning pattern. Covered by `AquaFlow.WebAPI.Tests` (see Runtime section).
- `WaterMetersController` (`/WaterMeters`) applies the same self-service read pinning for the `Customer` role (it still carries the `// TODO: add [RequirePermission(...)]` marker, so Admin/Collector gating is not defined yet): `GetAll`/`GetById` are overridden without a permission attribute. When the JWT `UserRole` claim is `Customer` (case-insensitive), `GetAll` force-pins `search.CustomerId` to the caller's `CustomerProfile.Id`, resolved from the JWT `Id` claim via the injected CustomerProfile `IBaseCRUDService` (`GetAllAsync` with `UserId` filter, same lookup the FE `ProfileService` uses over HTTP); a customer with no profile gets an empty page instead of the unfiltered listing. `GetById` returns 404 (not Forbid, to avoid confirming the id exists) when the meter's `CustomerId` doesn't match the caller's profile id - `WaterMeter.CustomerId` is a direct column (see the `WaterMeter`/`CustomerProfile` bullet below), so this check is a plain comparison, no Mapster flattening involved. A `Collector` (and `Admin`) passes through unfiltered - unlike `Customer`, a collector may read any meter, so `GetAll`/`GetById` never pin anything for that role. Covered by `AquaFlow.WebAPI.Tests/WaterMeters`. `WaterMeterSearchObject.Term` is a free-text OR search (`WaterMeterService.ApplyFilters`, lower-cased on both sides so it works under the InMemory test provider too, not just SQL Server's default collation) across `SerialNumber`, the owning `Customer`'s `FirstName`/`LastName`/concatenated full name, `Settlement.Name`, `Street`, and `HouseNumber` - on top of, not instead of, the existing `SerialNumber`/`SettlementId`/`Status`/`CustomerId` filters. `WaterMeterResponse` flattens `CustomerFirstName`/`CustomerLastName` from the linked `CustomerProfile` (same Mapster `NewConfig<WaterMeter, WaterMeterResponse>` pattern as `SettlementName`), which also requires `WaterMeterService.IncludeForRead`/`LoadReferencesAsync` to load the `Customer` navigation alongside `Settlement`. Covered by `AquaFlow.Services.Tests/WaterMeterServiceTests`.
- `SettlementsController` (`/Settlements`), `CitiesController` (`/Cities`), and `MunicipalitiesController` (`/Municipalities`) override `Create`/`Update`/`Patch`/`Delete` with `[RequirePermission("Locations.Manage")]` and have no read pinning of their own (the administrative lookup isn't owned by a customer). The gates are pinned by attribute-reflection tests in `AquaFlow.WebAPI.Tests` (`Settlements`/`Cities`/`Municipalities` folders). There is no `ServiceLocationsController`/`ServiceLocationService` anymore - see the `CustomerProfile`/`WaterMeter`/`FaultReport` bullet below for what replaced it.
- `TariffsController` (`/Tariffs`) overrides `Create`/`Update`/`Patch`/`Delete` with `[RequirePermission("Tariffs.Manage")]`, same pattern as `SettlementsController`; `GetAll`/`GetById` stay ungated (any authenticated caller can read tariffs). Pinned by an attribute-reflection test in `AquaFlow.WebAPI.Tests/Tariffs`.
- `MeterReadingsController` (`/MeterReadings`) is registered by hand (`MeterReadingService : EfCrudService<MeterReading, ...>`, not the generic `AddCrud<>`) because the collector data-entry flow needs more than plain CRUD. The generic `POST /MeterReadings` (base `Create`, still carrying the `// TODO: add [RequirePermission(...)]` marker for Update/Patch/Delete/GetAll/GetById) stays available for administrative backfill using the full `MeterReadingInsertRequest` shape. The real collector entry point is `POST /MeterReadings/collector-entry` (`CreateForCollector`) - see the "Meter Reading Flow" section below for the request shape, permission gating, and business rules.
- `WaterMeterRequestsController` (`/WaterMeterRequests`) is the shared customer/admin/collector surface for requesting and processing a new water meter. `Create` is overridden: `WaterMeterRequestInsertRequest` carries the requested address (`SettlementId`/`Street`/`HouseNumber`) plus an optional `Note`, but no `CustomerId`/`Status` - the controller reads the caller's user id from the JWT and calls `IWaterMeterRequestService.CreateForUserAsync`, which validates the supplied `SettlementId` exists (`ClientException`), resolves the caller's `CustomerProfile` (400 `ClientException` when there is none), and forces the initial status to `Pending`; the service's plain `InsertAsync` throws `ClientException` so a request can never be created under someone else's profile. `GetAll`/`GetById` stay without `[RequirePermission]`: a caller with `WaterMeterRequests.Manage` sees all requests, a `Collector` is pinned to their own `CollectorProfile.Id` via `AssignedCollectorId`, and a `Customer` is pinned to their own `CustomerProfile.Id`; `GetById` returns 404 for rows outside the caller's ownership. `POST {id}/cancel` is requester-only (404 when the request's `CustomerId` doesn't match the caller's profile, mirroring `UserNotificationsController.GetById`). `POST {id}/assign` and `POST {id}/reject` require `WaterMeterRequests.Manage`; assign also verifies the target `CollectorProfile` exists and its linked user is active. Generic `PUT`/`PATCH`/`DELETE` are also gated with `WaterMeterRequests.Manage`. `POST {id}/register` deliberately has no permission attribute: the caller must resolve to the exact `CollectorProfile.Id` stored in `AssignedCollectorId` or gets 404, then the state machine creates the real `WaterMeter` at the collector-supplied (possibly corrected) address and closes the request as `Registered`. `GET {id}/allowed-actions` applies the same manager/customer/collector ownership rules.
- `AccessController` (`/Access`) is anonymous: `POST /Access/login`, `POST /Access/refresh`, and `POST /Access/register` (public self-registration; always creates a `Customer`, ignoring any caller-supplied role). `UserRegisterRequest` also carries `FirstName`/`LastName` (`UserRegisterValidator` requires both, plus a 6-character password minimum); `Register` creates the `User` then immediately creates the matching `CustomerProfile` (same `CustomerProfileService`, auto-generated `CustomerCode`) so a self-registered customer always has a profile from the start, unlike admin-created users where the profile is optional. The response returned to the caller is the `UserResponse` captured before the profile insert, so its `FirstName`/`LastName` read as empty even though the profile was created correctly - only relevant if something other than the mobile client (which discards the body and re-logs-in) consumes this response. Registration does not return tokens, so the mobile FE (`AuthProvider.register` in `AquaFlow/UI/lib/shared/providers/auth_provider.dart`) follows up with `login` using the same credentials to establish a session.
- `AccountController` (`/Account`) is `[Authorize]` self-service for the signed-in user: `GET /Account/me`, `PUT /Account/me`, and `PUT /Account/me/password`. The user id always comes from the JWT `Id` claim (`ClaimNames.Id`), never from the request, so a caller can only ever read/edit their own record - which is why it needs no `Users.Manage` permission (the `UsersController` write actions do) and is safe for every role. `PUT /Account/me` takes `AccountUpdateRequest` (Email + Phone only) and calls `IUserService.UpdateOwnAccountAsync`, which updates only those two fields; role and active state are deliberately not self-editable anywhere (no privilege escalation). Password changes go through the separate `PUT /Account/me/password` instead: `AccountChangePasswordRequest` (CurrentPassword + NewPassword, validated by `AccountChangePasswordValidator` - both non-empty, NewPassword min 6 chars) is handled by `IUserService.ChangeOwnPasswordAsync`, which verifies `CurrentPassword` against the stored hash/salt via `ICryptoService.Verify` (throwing `ClientException` -> 400 on mismatch, so a stolen access token alone isn't enough to silently take over the password) before hashing and storing `NewPassword` with the existing `SetPassword` helper. `AccountUpdateValidator`/`AccountChangePasswordValidator` are registered in `Program.cs`.
- `AccessManager` validates credentials, issues the JWT (claims: id, email, userRole, isActive, plus one `Permission` claim per active permission of the user's role), and stores a refresh token (`RefreshToken` entity, 7-day expiry). Refresh rotates tokens. Permission codes for a role are resolved through `IPermissionLookupService` (`PermissionLookupService.GetPermissionCodesForRoleAsync`); keep that DB query in the service layer rather than in `AccessManager`.
- `Users` use soft delete. `DELETE /Users/{id}` (`UserService.DeleteAsync`) sets `IsDeleted=true`, `DeletedAt=UtcNow`, `IsActive=false`, and removes the user's refresh tokens; it does not remove `CustomerProfile`, `CollectorProfile`, or historical business rows. `UserService.GetDataSource`, account update/password paths, `GetByEmailAsync`, and `UpdateLastLoginAtAsync` all ignore soft-deleted users, so deleted accounts disappear from `/Users` and cannot log in or refresh. The `Users.Email` unique index is filtered to active rows (`IsDeleted = 0`), and `EnsureUniqueEmailAsync` follows the same rule, so an email can be reused after a soft delete.
- Refresh tokens are stored only as SHA-256 hashes (`RefreshTokenService.HashToken`): the raw value goes to the client, the DB row holds the hash, and lookups hash the incoming value before matching on the unique `RefreshToken.Token` index. `RefreshTokenService.InsertAsync` also purges expired tokens opportunistically. Do not persist raw refresh tokens.
- `/Access/login` and `/Access/refresh` are rate-limited (`RateLimitingPolicies.Authentication`, fixed window 5 requests/minute per client IP) via `AddRateLimiter`/`UseRateLimiter` in `Program.cs` and `[EnableRateLimiting]` on the actions.
- Every other request is throttled by a global limiter (`RateLimitingPolicies.Standard`, fixed window 300 requests/minute, set via `options.GlobalLimiter` in `Program.cs`), partitioned per authenticated user (JWT `Id` claim, available at this point since `UseAuthentication`/`UseAuthorization` run before `UseRateLimiter`) or per client IP for anonymous calls. A global limiter always stacks with any endpoint-specific policy rather than being overridden by it, so `/Access/login`/`/Access/refresh` still get the stricter 5/minute `Authentication` policy on top. This slows down scripted ID enumeration (e.g. `GET /UserNotifications/1`, `/2`, `/3`...) across all resource endpoints without needing a policy on every controller.
- Passwords are hashed with `ICryptoService` (PBKDF2 / `Rfc2898DeriveBytes`, SHA256, 10000 iterations, 20-byte hash, salt base64). Users store `PasswordHash` + `PasswordSalt`; never store plaintext.

Seed login credentials (local demo DB only; safe to use for testing): emails `admin@aquaflow.ba`, `collector@aquaflow.ba`, `customer@aquaflow.ba`, all with password `AquaFlow123!`.

To call a protected endpoint manually: `POST /Access/login` with `{"email":"admin@aquaflow.ba","password":"AquaFlow123!"}`, read `accessToken` from the response, then send `Authorization: Bearer <accessToken>`.

## Meter Reading Flow

The collector meter-reading flow spans `WaterMetersController`, `MeterReadingsController`, `BillingCyclesController`, and (as of the tariff/auto-invoice addition below) `Tariff`/`Invoice`/`InvoiceItem`:

- **Meter search**: `GET /WaterMeters?Term=...` (`WaterMeterSearchObject.Term`) OR-matches `SerialNumber`, the owning `Customer`'s `FirstName`/`LastName`/concatenated full name, `Settlement.Name`, `Street`, and `HouseNumber` (`WaterMeterService.ApplyFilters`, lower-cased on both sides so it works under the InMemory test provider too, not just SQL Server's default collation) - on top of, not instead of, the existing `SerialNumber`/`SettlementId`/`Status`/`CustomerId` filters. A `Collector` is not ownership-pinned (unlike a `Customer`), so this can find any meter.
- **Reading entry point**: `POST /MeterReadings/collector-entry` (`MeterReadingsController.CreateForCollector`, gated `[RequirePermission("MeterReadings.Manage")]` - seeded onto both the Admin and Collector roles) takes the deliberately reduced `MeterReadingCollectorEntryRequest` (`WaterMeterId`, `ReadingValue`, required `TariffId`, optional `Note`/`PhotoUrl`/`ClientUuid`/`BillingCycleId` - no `CollectorId`, same exclusion pattern as `WaterMeterRequestInsertRequest`). Returns `MeterReadingCollectorEntryResponse` (extends `MeterReadingResponse` with `InvoiceId`/`InvoiceNumber`/`InvoiceTotalAmount`, see below). The generic `POST /MeterReadings` (base `Create`, full `MeterReadingInsertRequest` shape, `TariffId` optional there) stays available for administrative backfill and does not auto-create an invoice.
- **CollectorId from JWT**: `IMeterReadingService.CreateForCollectorAsync` resolves `CollectorId` from the caller's JWT `Id` claim via `CollectorProfiles.UserId` - a client can never supply their own `CollectorId` (`ClientException` if the caller has no collector profile).
- **Tariff selection**: the collector picks the tariff to apply from the active tariff list (`GET /Tariffs?IsActive=true`) and sends its id as `TariffId`; the server re-validates it exists and `IsActive` (`ClientException` otherwise - "not found or is not active"). Multiple tariffs may be active at once (e.g. household vs. commercial) - there is no "single active tariff" rule like `BillingCycle`'s single-Open rule. The resolved `TariffId` is also pinned onto the `MeterReading` row itself, not just used for pricing.
- **Binding a reading to its `BillingCycle`**: an explicit `BillingCycleId` must exist and be `Open`; otherwise there must be exactly one `Open` cycle system-wide, else `ClientException` (zero or multiple open cycles).
- **Uniqueness per period**: a duplicate reading for the same `(WaterMeterId, BillingCycleId)` pair is rejected (`ClientException`; also enforced by a filtered unique DB index on `MeterReading` ignoring null `BillingCycleId`).
- **Lower-reading guard**: a `ReadingValue` lower than `WaterMeter.LastReading` is rejected unless the request carries a `Note` (treated as an explicit override for a meter replacement/reset).
- **Auto-generated Draft invoice**: on success, `CreateForCollectorAsync` sets `PreviousReadingValue`/`ConsumptionM3`/`ReadingDate` (`UtcNow`)/`Source` (`"Collector"`) on the reading and updates `WaterMeter.LastReading`, and in the SAME `SaveChangesAsync` also creates an `Invoice` (`Status = Draft`, `BillingCycleId`/`BillingPeriodFrom`/`BillingPeriodTo` from the resolved cycle, `Subtotal = TotalAmount = ConsumptionM3 * Tariff.PricePerM3` rounded to 2 decimals, `Tax = 0` - the `TaxRate` entity exists but is not wired up/seeded anywhere yet, `DueDate = null`, `CreatedById` = the collector's own user id) plus one `InvoiceItem` (`TariffId`, `Quantity = ConsumptionM3`, `UnitPrice = Tariff.PricePerM3`, `Amount = Subtotal`). `InvoiceNumber` is generated by a private year-scoped sequential helper (`GenerateInvoiceNumberAsync`, e.g. `INV-2026-0001`, resetting every calendar year - mirrors `CustomerProfileService.GenerateCustomerCodeAsync`/`CollectorProfileService.GenerateEmployeeCodeAsync`'s `StartsWith`+`AsSpan`+`Max()+1` pattern, just with a year-embedded prefix instead of a flat one). This is the only path that creates an invoice from a reading; the invoice is built directly against the `DbContext` (not through `IInvoiceService`) so it lands in the same transaction as the `MeterReading` insert and the `WaterMeter.LastReading` update. Note: Collector does not hold `Invoices.Read`/`Invoices.Manage` - this is deliberate, since the collector already gets the invoice id/number/total back inline in the response and has no other need to browse `/Invoices`.
- **`Invoice.DueDate` is nullable and unconstrained by design**: an invoice is not required to carry a payment deadline (the customer is not limited to a number of days to pay). `MarkOverdue` (`POST /Invoices/{id}/mark-overdue`) is a purely manual admin transition and never reads/compares `DueDate`, so leaving it `null` has no effect on the state machine.
- **BillingCycle admin path**: `BillingCyclesController` (`/BillingCycles`) is full CRUD - `BillingCycleService : EfCrudService<BillingCycle, BillingCycleResponse, BillingCycleSearchObject, BillingCycleInsertRequest, BillingCycleUpdateRequest, BillingCyclePatchRequest>`. `GetAll`/`GetById` stay ungated (any authenticated caller can read - a collector needs `GET /BillingCycles?Status=Open` for the current-period lookup, same as `TariffsController`); `Create`/`Update`/`Patch`/`Delete` require `[RequirePermission("BillingCycles.Manage")]` (seeded onto Admin only, not Collector). Only one `Open` cycle is allowed system-wide: `BeforeInsertAsync`/`BeforeUpdateAsync`/`BeforePatchAsync` throw `ClientException` when a request would set a row to `Open` while another row is already `Open`. Closing a cycle (`Status` `"Open"` -> `"Closed"`) auto-stamps `ClosedAt`; reopening one (`"Closed"` -> `"Open"`) clears it - neither request DTO exposes `ClosedAt` directly, so the service mutates the tracked entity in `BeforeUpdateAsync`/`BeforePatchAsync` before the Mapster mapping runs. `BillingCycleSearchObject.Status` filters via the generic reflection-based `ApplyFilters`. One `Open` `BillingCycle` (`Id=1`, "Juli 2026", 2026-07-01..2026-07-31) is seeded (`AquaFlowDbContextSeed.SeedBillingCycles`) so the collector-entry flow and the current-period lookup both have data out of the box; an Admin can now close it and open the next period through this endpoint instead of editing SQL by hand.
- Covered by `AquaFlow.Services.Tests/MeterReadingServiceTests` (happy path incl. auto-created Invoice/InvoiceItem assertions, invoice-number sequencing across two calls, inactive/unknown tariff rejection, duplicate, no/multiple open cycles, lower-reading rejection and its `Note` override, missing collector profile) + `AquaFlow.Services.Tests/BillingCycleServiceTests` (single-open-cycle invariant, close/reopen `ClosedAt` transitions, period validation), and `AquaFlow.WebAPI.Tests/MeterReadings` (JWT-sourced `CollectorId`, unauthenticated 401, permission-attribute pin) + `AquaFlow.WebAPI.Tests/BillingCycles` (permission-attribute pin on the write actions).

## UI (Flutter client)

**Admin desktop (`lib/admin/`) matches the current backend shape** - `ServiceLocation` is gone from the Dart side too (no `AdminServiceLocationsScreen`/`AdminServiceLocationService`/`AdminServiceLocation*` models, no "Lokacije" actions or drill-ins anywhere); the admin location lookup follows the City -> Municipality -> Settlement hierarchy (see the `AdminCodebookScreen` bullet below) and addresses live on `CustomerProfile`/`WaterMeter` as described in the `CustomerProfile`/`WaterMeter`/`FaultReport` bullet in "Architecture Rules" above.

**Mobile customer/collector (`lib/customer/`, `lib/collector/`) also matches the current backend shape** - the `/ServiceLocations` calls and `ServiceLocationId`/`CustomerServiceLocation`-shaped models are gone (no `customer_service_location.dart` anymore); see the "Vodomjeri"/"Nalozi" bullets below for what replaced them.

The Flutter client lives in `AquaFlow/UI` (package `aquaflow_desktop`, Dart `^3.12`) and targets desktop, mobile and web from one codebase. Dependencies: `http`, `provider`, `flutter_secure_storage`, `jwt_decoder`, `url_launcher`. See `AquaFlow/UI/README.md` for run instructions and the local-dev network notes.

- Folder layout is **feature-based**, not type-based. `lib/main.dart` is the entry point; `lib/app/` holds the routing layer (`platform_gate.dart`, `mobile_role_router.dart`, `unavailable_screen.dart`); `lib/shared/` holds all cross-cutting code (`config/`, `theme/`, `models/`, `services/`, `providers/`, and role-agnostic `screens/` including the reusable `mobile_shell.dart`); `lib/admin/`, `lib/customer/`, `lib/collector/` hold the per-role screens. The dependency direction is one-way: `admin`/`customer`/`collector` -> `shared` (never the reverse). Inside `lib/shared/` the sub-structure mirrors the old flat layout so files there keep relative imports (`../models/...`); anything crossing a feature boundary uses package imports (`package:aquaflow_desktop/...`).
- Routing (platform + role) is staged in `main.dart` + `lib/app/`: (1) **web block** - `kIsWeb` short-circuits to `UnavailableScreen` before login (the app is desktop/mobile only, not a web app); (2) **auth gate** - `_AuthGate` renders splash / `LoginScreen` / (authenticated) `PlatformGate` from `AuthProvider.status`; (3) **platform + role** - `PlatformGate` allows only `admin` on desktop (Windows/macOS/Linux, detected via `defaultTargetPlatform`) -> `AdminDashboardScreen`, and blocks any other role with an `UnavailableScreen` (+ logout); on mobile/tablet (Android/iOS) `MobileRoleRouter` maps `customer` -> `CustomerShell`, `collector`/`admin` -> `CollectorShell` (an admin on a phone has no dedicated UI, so by product decision reuses the collector shell), else `UnavailableScreen`. Roles are matched case-insensitively on `session.userRole`. `UnavailableScreen` (`lib/app/unavailable_screen.dart`) is the shared dead-end (icon + title + message + optional "Odjava" button; omitted for the web block since there is no session yet).
- Base URL lives only in `lib/shared/config/api_config.dart` (`ApiConfig.baseUrl`, port 5161). It picks the host per platform: `10.0.2.2` on the Android emulator, `localhost` on iOS simulator/desktop/web, and `ApiConfig.lanHostOverride` (a compile-time constant, default `null`) for a physical phone. Never hardcode the host anywhere else - always read `ApiConfig.baseUrl`.
- Auth flow: `AuthApiService` (`lib/shared/services/auth_api_service.dart`) calls `POST /Access/login`, `POST /Access/refresh`, and `POST /Access/register`, returning an `AuthResult` (`lib/shared/models/auth_result.dart`, the `{ accessToken, refreshToken }` pair) for login/refresh and mapping HTTP/transport failures to `AuthException`; `register` returns `void` (the backend response carries no tokens - see the `AccessController` bullet above) via a shared `_post` helper used by all three calls. `TokenStorage` (`flutter_secure_storage`) persists the access/refresh token pair. `AuthProvider` (a `ChangeNotifier`, `lib/shared/providers/auth_provider.dart`) is the single source of truth: `bootstrap()` restores a session on startup (valid access token, else silent refresh), `login()`/`logout()` mutate state, and `register()` calls `AuthApiService.register` then immediately `login()` with the same credentials so a new signup lands in an authenticated session in one flow. `AuthSession` (`lib/shared/models/auth_session.dart`) decodes the JWT via `jwt_decoder` using the backend claim names `Id`/`Email`/`UserRole`/`IsActive`/`Permission` (the `Permission` claim may decode to a string or a list; it is normalised to a list). `RegisterScreen` (`lib/shared/screens/register_screen.dart`) is the signup form (First/Last name, email, phone, password + confirm), pushed from a link on `LoginScreen`; on success it pops itself to reveal the root `_AuthGate`, which has already swapped to the authenticated flow underneath. See the Routing bullet above for how `main.dart` renders splash/login and hands an authenticated session to the platform/role gate.

## Architecture Rules

- Keep controllers in `AquaFlow.WebAPI/Controllers`; keep business logic in `AquaFlow.Services`; keep shared DTOs/contracts in `AquaFlow.Model`. Avoid business logic in controllers.
- Use `BaseReadController`/`BaseCRUDController` for standard endpoints.
- For standard CRUD, use the generic `EfCrudService<TEntity, TResponse, TSearch, TInsert, TUpdate, TPatch>`. Override its hooks instead of rewriting CRUD:
  - `IncludeForRead` / `IncludeForUpdate` for `.Include(...)` navigation loading.
  - `BeforeInsertAsync` / `BeforeUpdateAsync` / `BeforePatchAsync` for validation that needs the DB (uniqueness, reference existence). These run after FluentValidation and before entity mapping.
  - `LoadReferencesAsync` to load navigations after save (so the mapped response is populated).
- Use a resource-specific service (extending `EfCrudService`) when CRUD needs relationship wiring or uniqueness checks: see `PermissionService`, `UserRolePermissionService`, `CustomerProfileService`, `CollectorProfileService`, `TariffService`. `UserService` extends `BaseCRUDService` directly and overrides `InsertAsync`/`UpdateAsync`/`PatchAsync` because it also manages password hashing.
- `TariffService` replaced the generic `AddCrud<Tariff,...>()` registration (no navigations to load, so no `IncludeForRead`/`LoadReferencesAsync` overrides, unlike `SettlementService`). `EnsureUniqueNameAsync(name, int? excludedId = null)` is case-insensitive and global (no composite key, unlike `Settlement.Name`), called from `BeforeInsertAsync`/`BeforeUpdateAsync`/`BeforePatchAsync` (patch only when `request.Name != null`). `BeforePatchAsync` also re-validates the effective period when either `EffectiveFrom` or `EffectiveTo` is supplied, falling back to the entity's existing value for whichever half isn't in the request (`var from = request.EffectiveFrom ?? entity.EffectiveFrom; var to = request.EffectiveTo ?? entity.EffectiveTo;`), throwing `ClientException` if `to < from` - the same fallback pattern as `SettlementService.BeforePatchAsync`'s composite check. `TariffInsertValidator`/`TariffPatchValidator` additionally enforce `EffectiveTo >= EffectiveFrom` via FluentValidation when both values are present in the request (`TariffUpdateValidator` inherits the insert rule through `Include`), so a same-request violation is a 400 before the service layer even runs; the service-layer check in `BeforePatchAsync` exists because patch requests can supply only one side of the pair. `DeleteAsync` blocks (`ClientException`) when any `InvoiceItem.TariffId` references the row (that FK is `Restrict`, so the raw delete would fail anyway); otherwise hard-deletes. `TariffSearchObject.EffectiveOn` (a `DateTime?`) has no matching `Tariff` property name, so the generic reflection-based `BaseReadService.ApplyFilters` ignores it; `TariffService.ApplyFilters` overrides to call `base.ApplyFilters` first (for `Name`/`CustomerType`/`IsActive`) then, when `EffectiveOn` is set, adds `Where(t => t.EffectiveFrom <= date && (t.EffectiveTo == null || t.EffectiveTo >= date))` to find tariffs in effect on that date. Covered by `AquaFlow.Services.Tests/TariffServiceTests.cs`.
- `Invoice` uses a State pattern instead of generic CRUD. `InvoiceService` (`IInvoiceService`) extends `EfCrudService` but is registered by hand in `Program.cs` (not via `AddCrud`); the generic `IBaseCRUDService<InvoiceResponse,...>` alias still resolves to it. Status transitions live in `AquaFlow.Services/InvoiceStateMachine`: `BaseInvoiceState` is a pure state (no factory) plus one class per status: Draft/Issued/PartiallyPaid/Overdue/Paid/Cancelled. Each state is registered in `Program.cs` as a keyed scoped `BaseInvoiceState` with the `InvoiceStatus` constant as the key, and `IInvoiceStateResolver`/`InvoiceStateResolver` resolves the state for a status via `GetRequiredKeyedService<BaseInvoiceState>(status)` (throwing `ClientException` (400) for an unknown status). `InvoiceService` depends on `IInvoiceStateResolver`. Each public transition (`Issue`/`RecordPayment`/`Cancel`/`MarkOverdue`) loads the tracked `Invoice` once in `InvoiceService`, resolves the state from `entity.Status`, and passes that entity into the state action (state actions take an `Invoice`, not an `int id`), so a transition is a single read instead of a status query plus a full re-read. `GetAllowedActionsAsync` still uses a status-only projection since it never mutates. Every transition goes through `BaseInvoiceState.TransitionToAsync`, which flips `Invoice.Status` and writes an `InvoiceStatusHistory` row in one `SaveChanges`. Disallowed transitions throw `ClientException` (400). `RecordPaymentInternalAsync` runs its read-balance/insert-payment/transition sequence inside a `Serializable` transaction so concurrent payments cannot both pass the balance check and overpay the invoice; because the entity was preloaded outside that transaction, it calls `DbContext.Entry(invoice).ReloadAsync()` to bring the invoice-row read (and its range lock) back inside the transaction alongside the payments-sum read. A full payment always transitions to `Paid`; for a partial payment each state passes its own `partialStatus` target into `RecordPaymentInternalAsync` (Overdue stays `Overdue` so the overdue marker is not lost, Issued/PartiallyPaid land on `PartiallyPaid`), and the `InvoiceStatusHistory` note records the resulting status. New invoices are forced to `Draft` in `BeforeInsertAsync`. Each state declares the `InvoiceStatus` it represents through an abstract `Status` property (used by `BaseInvoiceState.NotAllowed` for the rejection message) and advertises its permitted actions via `GetAllowedActions()`, whose verbs come from the `InvoiceAction` constants (`Issue`/`RecordPayment`/`Cancel`/`MarkOverdue`, the clean API verbs without the `Async` suffix); that list is hand-maintained and must stay in sync with the transition methods each state actually overrides. `InvoicesController` exposes `POST {id}/issue|payments|cancel|mark-overdue` and `GET {id}/allowed-actions`. `InvoicesController` overrides `Create`/`Update`/`Patch`/`Delete` plus all five of the actions above (`Issue`/`RecordPayment`/`Cancel`/`MarkOverdue`/`GetAllowedActions`) with `[RequirePermission("Invoices.Manage")]` (`Permission.Id=13`, seeded onto Admin only via `UserRolePermission.Id=17`; Collector deliberately has neither `Invoices.Read` nor `Invoices.Manage`, per the "Auto-generated Draft invoice" bullet above). `Issue`/`RecordPayment`/`Cancel`/`MarkOverdue` no longer take a `changedById` query parameter: now that every one of them is behind `[Authorize]` + `[RequirePermission]`, the private `ResolveChangedById()` reads the acting user solely from the JWT `Id` claim (`ClientException` if it is somehow missing/unparsable) - a caller can no longer claim to act as a different user via the query string. `GetAll`/`GetById` are also overridden, gated `[RequirePermission("Invoices.Read", "Invoices.Manage")]` (either code is accepted - Customer holds `Invoices.Read` from the seed, Admin holds both, Collector holds neither and gets 403 at the filter before the action body runs). Inside the action: a caller holding `Invoices.Manage` (`HasManagePermission()`, a plain `Permission`-claim check, case-insensitive - not a role check) passes through unfiltered; otherwise `search.CustomerId`/the fetched row's `CustomerId` is pinned/compared against the caller's own `CustomerProfile.Id`, resolved from the JWT `Id` claim via an injected CustomerProfile `IBaseCRUDService` - identical mechanism to `WaterMetersController`'s role-based pinning, just keyed off the permission claim instead of `UserRole` since ownership here isn't tied to a single role. A customer with no profile gets an empty page from `GetAll`; `GetById` returns 404 (not Forbid) when the invoice's `CustomerId` doesn't match, to avoid confirming the id exists. `PaymentsController` overrides `Create`/`Update`/`Patch`/`Delete` with the same `[RequirePermission("Invoices.Manage")]` code (payments normally arise through `POST /Invoices/{id}/payments`; the generic write path here is administrative backfill only). Its `GetAll`/`GetById` are overridden the same way as `InvoicesController`'s, gated `[RequirePermission("Payments.Read", "Invoices.Manage")]` and pinning `PaymentSearchObject.CustomerId`/the fetched row's `CustomerId` (a direct column on `Payment`, covered by the generic reflection-based `ApplyFilters`) via the same `CustomerProfile`-resolution helper. `InvoiceItemsController` gates all six actions - including `GetAll`/`GetById` - with `[RequirePermission("Invoices.Manage")]`, the same "raw admin table, no self-service equivalent" precedent as `NotificationsController` (it has no per-customer ownership column to pin against anyway). All three are pinned by attribute-reflection tests plus (for Invoices/Payments) ownership-pinning tests in `AquaFlow.WebAPI.Tests/Invoices`, `Payments`, and `InvoiceItems`.
- `WaterMeterRequest` mirrors the Invoice state machine 1:1 in `AquaFlow.Services/WaterMeterRequestStateMachine/`: `BaseWaterMeterRequestState` (abstract `Status`, virtual `AssignAsync`/`RejectAsync`/`CancelAsync`/`RegisterAsync` that reject by default, `GetAllowedActions()` with verbs from `WaterMeterRequestAction`, `TransitionToAsync` writing a `WaterMeterRequestStatusHistory` row in the same `SaveChanges`) plus one state per `WaterMeterRequestStatus` constant (Pending/Assigned/Registered/Rejected/Cancelled, keyed scoped registrations in `Program.cs`, resolved via `IWaterMeterRequestStateResolver`). Pending allows Assign (sets `AssignedCollectorId`), Reject (reason goes into the history note) and Cancel; Assigned allows Register, which maps the passed `WaterMeterInsertRequest` to a real `WaterMeter` (its `CustomerId` always forced from the request entity's `CustomerId`; `SettlementId`/`Street`/`HouseNumber` come through `meterData` - the collector's on-site address, already validated server-side - see below), sets it as `ResultingWaterMeter` so EF assigns `ResultingWaterMeterId` in the same `SaveChanges`, and closes the request as Registered; Registered/Rejected/Cancelled are terminal. `WaterMeterRequestService` (`IWaterMeterRequestService`) is registered by hand like `InvoiceService` and adds `CreateForUserAsync` (see the controller bullet above) plus `AssignAsync`/`RejectAsync`/`CancelAsync`/`RegisterAsync`/`GetAllowedActionsAsync` transition methods that load the tracked entity once and delegate to the resolved state. `RegisterAsync` force-sets `meterData.CustomerId` from the request entity (security: the meter always belongs to the requester), then takes the new meter's `SettlementId`/`Street`/`HouseNumber` from `meterData` - the collector's (possibly corrected) address, which the FE prefills from the request's stored address - validating that the supplied `SettlementId` exists (`ClientException` otherwise, via `EnsureSettlementExistsAsync`) rather than deriving it from the customer's profile. It also copies the corrected `SettlementId`/`Street`/`HouseNumber` back onto the request entity so the stored request reflects what was registered, and never reads or modifies the customer's `CustomerProfile`. A collector's request body can never redirect the new meter to a different customer (CustomerId is forced), but it *can* correct the settlement/street/house number. `WaterMeterRequest` now carries its own full address: required `SettlementId` (FK `Settlement`, `Restrict`) + `Street` (MaxLength 200) + `HouseNumber` (MaxLength 30, string for "12A"/"bb"), supplied by the customer at creation. `CreateForUserAsync` validates the supplied `SettlementId` exists (`ClientException`, mirroring `CustomerProfileService`) and that the caller has a `CustomerProfile` (no per-location ownership/active checks - those disappeared with `ServiceLocation`, see the `CustomerProfile`/`WaterMeter`/`FaultReport` bullet below). Covered by `AquaFlow.Services.Tests/WaterMeterRequestServiceTests.cs`. `WaterMeterRequestUpdateRequest`/`PatchRequest` carry `SettlementId`/`Street`/`HouseNumber` alongside `Note` so an admin edit can adjust the address (patch validates each field only when provided); status and assignment fields still change exclusively through the state machine.
- The administrative location lookup is a three-level hierarchy: `City` (Name, Code) -> `Municipality` (Name, Code, required `CityId`) -> `Settlement` (Name, PostalCode, required `MunicipalityId`; the old free-form `Settlement.City` string is gone). `CityService`/`MunicipalityService`/`SettlementService` all extend `EfCrudService` and are registered by hand in `Program.cs` (`AddPatchMapping` + direct `AddScoped<IBaseCRUDService<...>>`, no `AddCrud<...>()`), sharing the same shape: case-insensitive `EnsureUnique<X>Async` checks in `BeforeInsertAsync`/`BeforeUpdateAsync`/`BeforePatchAsync` (patch only re-checks what the request actually touches, falling back to the entity's value for the other half of a composite pair), an `Ensure<Parent>ExistsAsync` FK check (`ClientException` when the parent id doesn't resolve; patch re-checks only when the FK is being set), and a `DeleteAsync` guard throwing `ClientException` naming what still references the row instead of hard-deleting (city -> municipalities; municipality -> settlements; settlement -> customer profiles/water meters/fault reports/water meter requests/collector profiles via `AssignedAreaId`/notifications); an unreferenced row still hard-deletes. Uniqueness rules: `City.Name` and `City.Code` globally unique; `Municipality.Name` unique within its city (composite `CityId`+`Name`), `Municipality.Code` globally unique; `Settlement.Name` unique within its municipality (composite `MunicipalityId`+`Name`). Each rule is backed by a matching unique index in `AquaFlowDbContext.OnModelCreating` (case-sensitive at the DB level - the app-level check is what actually catches a case-only duplicate). `MunicipalityResponse.CityName` and `SettlementResponse.MunicipalityName` flatten the parent's name via the Mapster config in `Program.cs` (`IncludeForRead`/`LoadReferencesAsync` load the parent navigation, same precedent as `CustomerProfileResponse.SettlementName`); `MunicipalitySearchObject.CityId` and `SettlementSearchObject.MunicipalityId` filter by parent (auto-applied by `BaseReadService.ApplyFilters` name matching). Covered by `AquaFlow.Services.Tests/CityServiceTests.cs`, `MunicipalityServiceTests.cs`, and `SettlementServiceTests.cs`. Seed: city "Sarajevo" (`SA`) pragmatically covers all nine Canton Sarajevo municipalities (`SA-01`..`SA-09`: Centar, Novi Grad, Novo Sarajevo, Stari Grad, Ilidza, Vogosca, Hadzici, Ilijas, Trnovo) with 2-3 real settlements each (Bjelave, Alipasino Polje, Dobrinja, Grbavica, Bascarsija, Hrasnica, ...); seed settlements 1/2 kept their ids (referenced by customer profile/water meter/fault report/collector area/notification seeds) and became Bjelave (Centar) and Hrasnica (Ilidza).
- There is no `ServiceLocation` entity. A customer's address lives directly on `CustomerProfile`: nullable `SettlementId` (FK `Settlement`) + nullable `Street`/`HouseNumber` (`HouseNumber` is a string, not an int, to allow values like `"12A"` or `"bb"`). These are nullable because a `CustomerProfile` is also created for admins/collectors who only need a name, not a service address. `CustomerProfileService.BeforeInsertAsync`/`BeforeUpdateAsync`/`BeforePatchAsync` throw `ClientException` when a supplied `SettlementId` doesn't resolve to an existing `Settlement` (only checked `.When(...HasValue)`, same convention as everywhere else); `IncludeForRead`/`LoadReferencesAsync` load `Settlement` so the Mapster config in `Program.cs` flattens `CustomerProfileResponse.SettlementName`. `CustomerProfileSearchObject.SettlementId` filters via the generic reflection-based `ApplyFilters` (no override needed). `WaterMeter` and `FaultReport` each carry their own required `CustomerId` (FK `CustomerProfile`) + `SettlementId` (FK `Settlement`) directly instead of a shared location row - `WaterMeterResponse`/`FaultReportResponse.CustomerId` are therefore plain columns, not Mapster flattening (`WaterMeterResponse.SettlementName` is still flattened, the same way `CustomerProfileResponse.SettlementName` is; `WaterMeterService.ApplyFilters` compares `SettlementId`/`CustomerId` directly instead of going through a location). `WaterMeter` additionally has nullable `Street` (MaxLength 200) + `HouseNumber` (MaxLength 30, string) columns - nullable so seeded meters that only had a settlement stay valid, and so a customer can have meters at different street addresses; they flow straight through as plain `WaterMeterResponse` columns (no flatten) and are populated when a meter is registered from a request that carried a full address. A water meter request now carries its own full address too: `WaterMeterRequest` has required `SettlementId` (FK `Settlement`) + `Street` + `HouseNumber`, supplied by the customer at creation and correctable by the assigned collector at registration (see the state machine bullet above for how `RegisterAsync` takes the meter's address from the collector's request body, forcing only the `CustomerId`). Registering a fault report requires the caller to supply both `CustomerId` and `SettlementId` directly (no location-existence check layer - that responsibility moved to whichever endpoint creates the `FaultReport`, currently the generic CRUD controller). `SettlementService.DeleteAsync` blocks deletion (`ClientException` naming the blocker) when the settlement is still referenced by any `CustomerProfile`, `WaterMeter`, `FaultReport`, or `WaterMeterRequest`, on top of the pre-existing `CollectorProfile`/`Notification` checks - those FKs are `Restrict`, so the raw delete would fail anyway.
- Uniqueness convention: a private `EnsureUnique<X>Async(value, int? excludedId = null)` that throws `ClientException($"... already exists.")` when a conflicting row exists (`... && entity.Id != excludedId`). Call it before mapping; pass `excludedId: null` on insert and the entity id on update/patch. Examples: `UserService.EnsureUniqueEmailAsync`, `PermissionService.EnsureUniqueCodeAsync`, `UserRolePermissionService.EnsureUniqueAssignmentAsync`, `CityService`/`MunicipalityService`'s `EnsureUniqueNameAsync`/`EnsureUniqueCodeAsync`, `SettlementService.EnsureUniqueNameAsync` (composite `MunicipalityId` + `Name`, case-insensitive), `TariffService.EnsureUniqueNameAsync` (globally unique, case-insensitive). Always back uniqueness with a DB unique index too.
- Throw `ClientException` for expected client errors (400), `KeyNotFoundException` for missing resources (404). `BaseCRUDController` and `ExceptionFilter` translate these (e.g. `ClientException` -> 400 `{ message, errors }`, FK violation on delete -> 400).
- Use FluentValidation validators in `AquaFlow.Services/Validators`; register each in `Program.cs`.
- Use Mapster for DTO/entity mapping. Patch requests map with `IgnoreNullValues(true)` via the `AddPatchMapping` helper.
- Keep EF Core model configuration in `AquaFlowDbContext` partials under `AquaFlow.Services/Database`.
- Keep connection strings and JWT secrets out of committed `appsettings*.json`; use environment variables or user secrets.

## API Conventions

- List endpoints (`GET /<Resource>`) return `PageResult<T>` = `{ "items": [...], "totalCount": <int?> }`. Paging/search comes from `BaseSearchObject`: `Page` (default 1), `PageSize` (default 10), `IncludeTotalCount` (default false), `SortBy` (entity property name, null = no sort), `SortDescending` (default false), plus resource-specific filters (e.g. `Users?Email=...&UserRoleId=...&UserRole=...&IsActive=...&Name=...`). `Users?Name=` matches (Contains) against the linked `CustomerProfile`'s `FirstName` OR `LastName` - a user with no `CustomerProfile` (admin/collector, or a customer with no profile yet) never matches it. `UserResponse.FirstName`/`LastName` are likewise flattened from `CustomerProfile` in the Mapster config in `Program.cs` (empty string when there is none); `UserService.GetDataSource()` includes `CustomerProfile` for both.
- Sorting in `BaseReadService.ApplySorting` is applied before paging using an Expression-tree key selector (no string-based/dynamic LINQ, no injection risk). An unknown `SortBy` is ignored (same lenient behaviour as filtering), and `SortBy` is matched case-insensitively against entity property names. To restrict which columns a resource can sort by, override `protected virtual HashSet<string>? SortableProperties` and return a whitelist; null (default) allows any existing entity property.
- Standard CRUD verbs per resource: `GET` (list), `GET /{id}`, `POST`, `PUT /{id}`, `PATCH /{id}`, `DELETE /{id}`.

## File Style

- One public type per `.cs` file. Do not group multiple entities, requests, responses, search objects, validators, interfaces, or controllers in one file.
- File name matches the public type, e.g. `WaterMeter.cs`, `WaterMeterResponse.cs`, `WaterMeterInsertValidator.cs`.
- Keep namespaces aligned with folders.
- Do not reintroduce generic template files like `Class1.cs`.

## Adding A New Resource

1. Add entity in `AquaFlow.Services/Database/{Name}.cs` (inherit `EntityBase`).
2. Add request DTOs in `AquaFlow.Model/Requests/{Name}InsertRequest.cs`, `{Name}UpdateRequest.cs`, `{Name}PatchRequest.cs`.
3. Add response DTO in `AquaFlow.Model/Responses/{Name}Response.cs` (inherit `AuditableResponse`).
4. Add search object in `AquaFlow.Model/SearchObjects/{Name}SearchObject.cs` (inherit `BaseSearchObject`).
5. Add validators in `AquaFlow.Services/Validators/{Name}InsertValidator.cs`, `{Name}UpdateValidator.cs`, `{Name}PatchValidator.cs`.
6. Add controller in `AquaFlow.WebAPI/Controllers/{Names}Controller.cs` (derive from `BaseCRUDController`).
7. Add the `DbSet<{Name}>` to `AquaFlowDbContext` and any model configuration in a `AquaFlowDbContext` partial.
8. Add EF seed data in `AquaFlowDbContextSeed.cs` if the SQL database should contain initial rows.
9. Register in `Program.cs`: simple resources use `AddCrud<...>()`; resources needing relationship wiring/uniqueness register a resource-specific `EfCrudService` subclass. Register all three validators.
10. Add and apply an EF Core migration if entity shape or SQL seed data changed.
11. Run `dotnet build .\AquaFlow\AquaFlow.sln`.
